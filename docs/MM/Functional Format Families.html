Functional Format Families

Functional Format Families
1 Audiovisual Source Language Data

[Audio|Video]/[Archive|Tool|Web+Distribution] 

Should be open and parameters for codecs, frame and bit rates etc. according to purpose. The distribution version might be an edited version derived from several primary recordings. 


 
2 Image/Legacy Textual Source Language Data

mages: unparsed legacy source documents, i.e. digitized as scans, possible with first automatic OCR, sometimes including legacy annotation/comments, i.e. markup, could also be annotated with polygon selections within a linguistic resource, included for reference in the same way as audiovisual sources. 

3 Textual Source Language Data
PlainText

only running, single stream of tokens, (written) text as is, no mark-up 

StructuredText

non-linguistic structured stuff like HTML or LaTeX e.g. for research on CMC, Wikipedia or academic writing 

4 Textual Language Data

Derived from a source. 

Should be open, documented, processable, checkable=have machine-readable conventions, … Maybe the QUEST distinction Deposit-Collection-Corpus could be useful. 

PlainText

only running, single stream of tokens, (written) text as is, no mark-up, nothing 

TextAnnotation

various forms of annotated text (which might also be non-linguistically structured), including semi-structured text with interspersed markup (text_N etc.), structured text formats like CSV/TSV, structured (often XML) annotation formats like TEI, stand-off formats leaving out the primary text etc., for now, we don’t recommend anything below annotation, e.g parallel corpus/translation memory formats, IGT formats or scholarly editions, since we can still recommend features of the formats even if exact formats (i.e. the annotation) change, and we can recommend to use existing formats, if necessary with adaptations -> resource families 

AudioVisualAnnotation

usually time-aligned, usually containing a textual representation of non-written language, often containing other information as annotation, here we can make recommendations for specific formats based on basic characteristics of spoken/signed language but also no further groups like language acquisition etc.

5 Contextual Data

Information on the language data and its creators (participants or authors) that makes analysis possible, hence should be processable/relatable, is sometimes contained (redundantly) in the annotation files. 

6 Contextual Source Data

Not the language data per se, but other data objects belonging to the data mainly relevant for multimodal corpora, e.g. texts used in a recorded conversation, images such as photos or drawings of the communicative situation - these are sources in the CAQDAS scene but usually somewhat neglected in linguistic research and (therefore) usually not processable. Textual data could of course be made available as language data, e.g. the text of a consent form being filled out could be included into a corpus of doctor-patient communication. 

7 Lexical (Conceptual) Resource
wordList
lexicon
wordNet
8 Specifications/Settings/Schemata/Vocabularies/Conventions

needs a name! 

Markup/annotation (but also tool) related stuff required for reliable reuse and processing. 


 
9 Geodata

Information on geographic locations 

10 Statistical Data

Data from surveys, GJTs etc., but not modelled as contextual data, or statistics on the resource not modelled as annotations or applying to the whole resource but not modelled as catalogue metadata - for reuse purposes. 

11 Catalogue Metadata

Structured resource documentation for discoverability and basic description. 

12 Resource Documentation

Unstructured documentation of the resource and its parts 

13 Query Engine Input/Tool+Service Format

Formats that exist only for distribution via query platforms, internal tool formats etc.
